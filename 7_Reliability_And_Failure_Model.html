01
GPU 장애의 "정상 상태(Normal State)" 이해
클러스터 규모가 커질수록 하드웨어 장애는 확률적으로 피할 수 없는 확정 이벤트가 됩니다.

대규모 GPU 클러스터에서 신뢰성 설계의 출발점은 "장애는 언제든 발생한다"는 전제를 수용하는 것입니다. 단일 컴포넌트의 MTBF(Mean Time Between Failures)가 아무리 높아도, 시스템 전체의 MTBF는 구성 요소 수에 반비례하여 급격히 낮아집니다.

Cluster MTBF = Single GPU MTBF ÷ Total GPU Count
단일 GPU MTBF = 26,280시간(약 3년) 가정 시, 1,000대 클러스터의 클러스터 MTBF ≈ 26.3시간 → 하루 1건 이상 장애 필연적 발생

02
NVIDIA XID Error 유형 및 원인 분석
NVIDIA 드라이버가 기록하는 XID Error 코드를 기반으로 하드웨어/소프트웨어 결함을 분류하고 자동 복구 판단 근거를 마련합니다.

XID(eXtended ID) 에러는 NVIDIA 드라이버가 /var/log/syslog 또는 dmesg에 기록하는 GPU 관련 이벤트 코드입니다. 단순 로그가 아니라 자동 복구 시스템의 판단 트리거로 활용됩니다. DCGM(Data Center GPU Manager)을 통해 실시간으로 수집됩니다. 

# /var/log/syslog 예시 — XID 에러가 감지된 실제 로그 패턴
NVRM: Xid (PCI:0000:3b:00): 79, pid='<unknown>', name=<unknown>, GPU-PCI-fallen-off-the-bus
NVRM: Xid (PCI:0000:3b:00): 48, pid='12304', name=python3, Double Bit ECC Error
NVRM: Xid (PCI:0000:3b:00): 13, pid='12304', name=python3, Graphics Exception: ESR 0x504200

XID 분류 요약표

에러 그룹 대표 XID 심각도 현상 및 주원인 서비스 영향 복구 전략 자동화 가능
소프트 메모리 오류 XID 13 Soft 허용되지 않은 메모리 주소 접근, CUDA 버그, 잘못된 포인터 연산 해당 프로세스만 중단 파드/프로세스 재시작 ✓ 자동
소프트 커널 오류 XID 31 Soft GPU 파이버 스케줄링 오류, 드라이버 내부 타임아웃 해당 컨텍스트 크래시 파드 재시작, 드라이버 재로드 고려 ✓ 자동
GPU 응답 없음 XID 43 Soft 무한 루프, Deadlock, 커널 실행 타임아웃 (>2min) 작업 블로킹, 노드 부분 응답 불능 파드 재시작 후 노드 헬스 재확인 ✓ 자동
DBE(더블 비트 ECC) XID 48 Hard VRAM 물리적 손상, 두 비트 동시 오류로 데이터 복구 불가 메모리 데이터 오염, 연산 결과 오류 노드 Taint → 스케줄링 배제 → 교체 ⚠ 반자동
버스 탈락 XID 79 Hard PCIe/NVLink 링크 손실, GPU가 버스에서 완전 이탈 (전원/보드 결함) 노드 전체 불안정, OS 크래시 가능 노드 즉시 격리 → 점검 → 교체 ⚠ 반자동
NVLink 패킷 손상 XID 119 Warning NVLink 물리 인터커넥트 오류, 패킷 체크섬 불일치 (과열, 접촉 불량) GPU 간 통신 성능 저하 또는 중단 재시작 시도 → 지속 시 노드 교체 ✓ 자동
NVLink 링크 단절 XID 120 Warning NVLink 링크 완전 단절, 119 이후 악화 단계 멀티 GPU 학습 불가 해당 GPU 격리 → 팬아웃 재구성 ✓ 자동
ECC 격리 성공 XID 94 Info Ampere 이상 아키텍처: SBE(Single-Bit) ECC 발생, 하드웨어가 성공적으로 격리 해당 작업만 중단, 노드 계속 사용 가능 작업 Kill + 재스케줄링, 노드는 유지 ✓ 자동
ECC 격리 실패 XID 95 Hard ECC 오류가 격리 범위를 벗어나 확산 — 노드 전체 오염 위험 노드 전체 불안정, 데이터 신뢰 불가 노드 즉시 격리 → 전체 메모리 검사 ⚠ 반자동

주요 XID 에러 상세 카드

79
GPU-PCI: Fallen off the bus
GPU가 PCIe 버스에서 완전히 이탈한 상태. 전원 공급 불안정, 보드 물리 결함, 또는 극심한 과열이 원인입니다. OS가 GPU 디바이스를 더 이상 인식하지 못하며, nvidia-smi에서 해당 GPU가 사라집니다. 재부팅 후에도 재현되면 하드웨어 교체가 필요합니다.
노드 즉시 Taint → 물리 점검 → 교체 판정
48
Double Bit ECC Error (DBE)
VRAM에서 두 개의 비트가 동시에 오류 — ECC로 복구 불가능한 수준의 물리 손상입니다. SBE(Single-Bit Error, XID 63)는 ECC가 자동 수정하지만, DBE는 데이터 오염이 확정됩니다. 방치 시 연산 결과 오류가 조용히 전파될 수 있습니다(Silent Data Corruption).
노드 Taint + 스케줄링 배제 → VRAM 교체
43
GPU Preemption Context Switch Timeout
GPU 커널이 2분(기본값) 이상 CPU로 제어권을 반환하지 않을 때 발생합니다. 무한 루프, Deadlock, 또는 지나치게 긴 커널 실행이 원인입니다. 재시작 후 재발 시 소프트웨어 코드 문제일 가능성이 높습니다.
프로세스/파드 재시작 → 재발 시 코드 검토
119/120
NVLink Error / Link Down
119는 NVLink 패킷 레벨 오류(CRC 불일치), 120은 링크 완전 단절입니다. 멀티 GPU 학습(DDP, Tensor Parallel)에서 All-Reduce 통신이 이 링크를 거치므로 발생 시 전체 학습 Job이 블로킹됩니다. 열 관리 및 케이블 점검이 우선입니다.
재시작 → 지속 시 해당 GPU 격리 → 점검
94
ECC Error — Contained (Ampere+)
Ampere 아키텍처(A100, H100)부터 도입된 Row Remapping 기능으로 ECC 오류를 하드웨어 수준에서 격리 성공. 노드 전체를 버리지 않고 해당 작업만 종료하면 됩니다. 격리에 성공했다는 신호이므로 즉각적인 교체보다는 지속 모니터링이 적절합니다.
작업 Kill → 재스케줄링 → 노드 모니터링 유지
13
Graphics Exception — Illegal Memory Access
CUDA 애플리케이션이 잘못된 메모리 주소에 접근할 때 발생합니다. 대부분 소프트웨어 버그이며, 하드웨어 이상이 아닙니다. cuda-memcheck 또는 compute-sanitizer를 통해 소프트웨어 레벨에서 디버깅 가능합니다.
파드 재시작 → 반복 시 코드 디버깅 필요
⚠ Silent Data Corruption (SDC) 주의 XID 48(DBE) 및 XID 95는 Silent Data Corruption(SDC)으로 이어질 수 있습니다. SDC는 연산 결과에 조용히 오류 비트가 삽입되어도 시스템이 크래시하지 않고 계속 동작합니다. 대규모 LLM 학습에서 SDC는 며칠 치 학습 결과를 무효화할 수 있으므로, DBE/ECC 관련 XID는 단순 재시작이 아닌 즉시 노드 격리가 원칙입니다.
03
Partial Failure 대응 및 복구 판단
멀티 GPU 노드에서 일부 GPU에만 결함이 발생했을 때, 노드 전체를 버리지 않고 가용 자원을 활용하는 전략입니다.

8-GPU 노드에서 1개 GPU만 XID 오류를 낸 경우, 노드 전체를 NoSchedule로 Taint하면 7개의 정상 GPU가 낭비됩니다. 오케스트레이터는 XID 유형에 따라 "제자리 재시작(Restart in Place)"과 "타 노드 재할당(Reallocation/Eviction)" 중 하나를 자동 선택해야 합니다.

주요 위험 하드웨어 결함일 경우 CrashLoopBackOff 반복 → 무한 재시작 루프 100GB+ 컨테이너 이미지 + 수백 GB 모델 가중치 Pull 시 네트워크 병목 발생
노드 상태 노드 스케줄링 유지 (해당 파드만 재시작) 노드에 Taint 부여 → 신규 파드 스케줄링 불가 → 점검 후 Taint 해제
DDP 환경 처리 단일 파드 재시작 시 나머지 워커 All-Reduce 대기 상태 → 타임아웃 필요 Gang Scheduling과 연동하여 전체 Job 일괄 재시작 (Volcano/Kueue)
실패 에스컬레이션 N회 재시작 실패 시 → 자동으로 Reallocation 전환 건강한 노드 부족 시 → 대기 큐(Pending) → 용량 확보 후 스케줄링
✅ Restart in Place가 유리한 상황
XID 로그가 소프트웨어/CUDA 버그를 명확히 가리킬 때
노드가 이전에도 안정적으로 동작한 이력이 있을 때
모델 가중치/체크포인트가 로컬에 캐시되어 있을 때
인퍼런스(Inference) 서빙처럼 짧은 다운타임이 중요할 때
클러스터에 여분 노드가 없을 때 (리소스 부족)
⚠️ Reallocation이 반드시 필요한 상황
XID 48/79/95 — 하드웨어 손상이 확인된 경우
재시작 후에도 동일 XID가 즉시 재발할 때
대규모 LLM 훈련 중 Silent Data Corruption이 의심될 때
DDP 환경에서 노드 1개 오류로 전체 Job이 블로킹될 때
누적 ECC 카운터가 임계치를 초과한 노드
04
통합 자동 복구 파이프라인 아키텍처
장애 감지부터 재스케줄링까지 사람의 개입 없이 오케스트레이션 레벨에서 처리하는 파이프라인입니다.

단순한 알림(Alert) 시스템이 아닌, 자율 치유(Self-Healing) 파이프라인을 구성해야 합니다. 각 단계는 느슨하게 결합(Loosely Coupled)되어 있어 하나의 컴포넌트 장애가 전체 복구 파이프라인을 마비시키지 않아야 합니다.

1
Detection 감지 — DCGM Exporter
NVIDIA DCGM(Data Center GPU Manager)의 Exporter가 각 노드에 DaemonSet으로 배포되어 GPU 메트릭을 밀리초 단위로 수집합니다. XID 에러, ECC 카운터, GPU 온도, 전력, VRAM 사용량, PCIe 대역폭, NVLink 오류 카운터 등을 Prometheus 형식으로 노출합니다.
주요 메트릭: DCGM_FI_DEV_XID_ERRORS · DCGM_FI_DEV_ECC_DBE_VOL_TOTAL
수집 간격: 1~5초 (권장) — 너무 짧으면 카디널리티 폭발
배포 방식: K8s DaemonSet (노드당 1 Pod) + nvidia-device-plugin 연동
2
Evaluation 평가 — Prometheus + Alertmanager
Prometheus는 수집된 메트릭에 룰셋(Recording Rules / Alerting Rules)을 적용합니다. XID 값의 심각도 분류, 에러 빈도 집계, 임계치 비교를 통해 Soft/Hard Error를 자동 판별합니다. Alertmanager는 중복 제거, 그룹화, 라우팅 후 다음 단계(격리)로 알림을 전달합니다.
Rule 예시: expr: DCGM_FI_DEV_XID_ERRORS{xid="79"} > 0
Routing: Hard Error → 즉시 격리 웹훅 / Soft Error → 재시작 웹훅
억제(Inhibit): XID 79 발생 시 동일 노드의 다른 알림 억제
3
Isolation 격리 — Node Problem Detector + Custom Controller
Node Problem Detector(NPD)는 노드 로컬에서 syslog를 파싱하여 XID를 감지하고 K8s API 서버에 NodeCondition을 업데이트합니다. Custom Controller(또는 Alertmanager 웹훅)는 결함 노드에 Taint를 자동으로 부여합니다. 동시에 Device Plugin이 갱신되어 결함 GPU만 가용 자원 풀에서 즉시 제외됩니다.
Taint 예시: kubectl taint node gpu-node-07 nvidia.com/gpu-fault=true:NoExecute
Device Plugin: /etc/kubernetes/node-feature-discovery → GPU UUID 기반 격리
부분 격리: 8GPU 노드에서 1GPU만 결함 → 해당 UUID만 할당 불가 처리
4
Rescheduling 재할당 — K8s Scheduler / Gang Scheduler
Taint된 노드에서 NoExecute Toleration 없는 파드가 자동으로 방출(Eviction)됩니다. K8s 스케줄러는 건강한 노드를 선택하여 워크로드를 재배치합니다. 분산 학습(DDP) 환경에서는 Volcano/Kueue 같은 배치 스케줄러와 연동한 Gang Scheduling으로 전체 Job을 일괄 재시작합니다.
단일 파드: K8s 기본 Scheduler → Healthy 노드 재배치
분산 학습: Volcano PodGroup / Kueue Workload → minAvailable 미달 시 Job 전체 재시작
체크포인트: 학습 중단 전 마지막 체크포인트로 롤백 (PyTorch Fault Tolerance)

💡 Gang Scheduling의 핵심 가치 분산 학습(DDP, FSDP, Tensor Parallel)에서 파드 하나라도 All-Reduce 통신에서 이탈하면 나머지 정상 파드들이 무한 대기(Deadlock)에 빠집니다. Gang Scheduling은 minAvailable 조건 미달 시 전체 Job을 즉시 철회(Preempt)하고 건강한 노드로 일괄 재할당하여 고가 GPU 자원의 공회전(Idle Waste)을 방지합니다. 1,000달러짜리 GPU 수십 대가 Deadlock 상태로 수 시간을 낭비하는 상황을 막는 것이 핵심입니다.

🗺 체크포인팅과 복구 시간 최적화

전략 설명 체크포인트 간격 복구 소요 시간 스토리지 오버헤드
로컬 SSD 체크포인트 노드 로컬 NVMe에 저장. 재할당 시 재사용 불가 15~30분마다 ~수 분 (같은 노드 복구 시) 낮음
공유 파일시스템 (NFS/Lustre) 모든 노드에서 접근 가능한 공유 스토리지에 저장 5~15분마다 ~수 분~10분 중간
오브젝트 스토리지 (S3/GCS) 내구성 높은 분산 오브젝트 스토리지에 저장 30~60분마다 ~수십 분 (다운로드 시간 포함) 낮음 (비용 효율)
인메모리 체크포인트 (NCCL) 다른 노드의 GPU 메모리에 미러링 (PyTorch 2.0+) 매 스텝 또는 N 스텝 ~수 초 높음 (VRAM 추가 점유)
📌 실무 권장 체크포인트 전략 대규모 LLM 훈련에서는 계층적 체크포인팅이 효과적입니다: ① 매 N 스텝마다 로컬 NVMe에 빠른 저장, ② 매 M 스텝(M > N)마다 공유 파일시스템 또는 오브젝트 스토리지에 영구 저장. PyTorch의 torch.distributed.checkpoint 및 torchsnapshot 라이브러리, 또는 Megatron-LM의 체크포인트 기능을 활용하면 저장/복구 시간을 최소화할 수 있습니다.
